# Workflow: UI Flow To Component Plan

## Purpose
Translate user flow and interaction requirements into screens, components, state, and fetch strategy.

## Trigger
Use when the design problem is bigger than a single component but smaller than whole-app architecture.

## Inputs
- user flow
- target surfaces
- data requirements

## Steps
1. Map the journey and interaction states.
2. Define screens and component responsibilities.
3. Plan responsive behavior, accessibility, and design-system alignment.
4. Plan state ownership and API/data-fetch integration.

## Outputs
- screen plan
- component plan
- state and data-fetch notes

## Review checkpoints
- primary flow stays simple
- components are reusable enough but not overabstracted
- failure and loading states are covered
